需求分析

1 需求的定义

1.1 什么是需求

需求是为了满足某类用户的某种欲望或解决某些问题时所产生的现象

1.2 需求的层次

马斯洛需求层次理论

生理需求>安全需求>社交需求>尊重需求>自我实现

生理需求：饿了么
安全需求：360
社交需求：QQ，微信
尊重需求：知乎
自我实现：抖音

1.3 刚需/痛点，痒点

  • 痛点对应的就是解决核心的问题，痛点就是刚需
  • 痒点就是满足欲望，是消费升级的副产品

1.4 需求的分类

需求按分类分为内部需求和外部需求

内部需求 外部需求
用户需求 客户/合作需求
运营需求 市场需求
技术需求
数据需求

  1. 用户需求
    解决或满足用户在某些场景下为了达到某些目的所产生的需求

  2. 运营需求
    满足公司内部运营对某类事件或活动所产生的推广等性质的需求

  3. 技术需求
    代码重构，系统安全等，技术人员提出的需求

  4. 数据需求

  • 基本数据
  • 用户数据
  • 点击埋点
  • 路径漏斗

数据知道产品，需要做到问道任何数据，均可以流畅的回答

  1. 客户需求/合作需求

需要平衡商业展示和用户体验
（比如：观看视频之前的广告，有一定的收入但是影响用户体验）

  1. 市场/战略需求

战略需求往往代表着市场的前沿，具有一定的前瞻性与风险

1.5 需求的来源

  1. 用户访谈

主要通过用户调研和市场调研

定性|定量
面对面访谈|问卷调查
可用性测试|PUSH
|有偿调查

定性走开放性问题，定量走简单可选择的问题
观察>引导，深挖但不被误导

  1. 数据分析
    通过基础的数据分析辅佐你想要优化的方向，是做需求的核心目的之一。
    • 基本数据
      • Crash
      • DAU/MAU
      • 留存
    • 用户数据
      • 性别
      • 地狱
      • 标签（包括购买率，页面流失情况，功能使用情况，产品情况）
    • 点击埋点
      • 点击次数
      • 点击人数
    • 路径漏斗
      • 页面层级
      • 页面流失
  2. 竞品分析
  • 选择相当的竞品
  • 功能颗粒度足够细
  • 长期维护（带版本）
  • 发觉竞品路线
  • 学会标注重点
  1. 用户反馈
    用户反馈主要通过用户评论
    用户反馈还可以通过微博，QQ/微信群，贴吧，亲朋好友，各类应用市场获取
  2. 头脑风暴
  • 3-8人
  • 定义一个主题
  • 提前准备

1.6 四要素比对

需求提炼中注意4个要素

  1. 人物
  2. 场景
  3. 任务
  4. 目标

案例：建议能将运动中拍到的照片传到运动的轨迹中去，便于查询存档。

人物 场景 任务 目标
用户属性 运动结束后 上传照片到轨迹 查询，存档？
运动属性
一个爱运动的女性
人物 场景 任务 目标
使用者&决策者 用户使用场景vs产品提供场景 多样性 用户的目标vs我们的目标
人群：宽&窄 复杂程度
用户属性

习惯性的对每个需求都拆分成4个结构，并对其进行分析比对，然后筛选出我们真正的需求。

1.7 需求的转化

2 需求转化

我们需要把来自各方的需求转化为我们产品的真正需求

两个例子

  1. 用户需求：希望有一个搜索评论的功能，有些歌忘了名字，但是还记得评论

2.用户需求：增加学习时长分享功能，统计的学习时长可以让用户进行“晒”

2.1 需求表达常见的问题

问题 思考不够全面 关联性不够清晰 复杂逻辑梳理困难 文字表达不直观 产品，研发，测试缺少统一参照标准
解决办法 发散思维 层级关系，俯视地图 梳理逻辑关联 图形化表达 归档
解决工具 思维导图 产品结构图，功能结构图 功能/页面流程图 交互原型图 需求文档
生命周期 分析调研 梳理思维 归纳逻辑 可视化 文档化
工具推荐 纸笔/XMind XMind XMind Axure Word

接下来我们来详细看一下这些解决办法

2.2 思维导图

有一点儿想法就要记录在思维导图上，大而全，我们以后会进行细致赛选

3 需求管理

需求管理主要有一下几个部分组成

  1. 需求池管理
    • 明确接口人
    • 明确时间
    • 保持更新
    • 保持反馈
  2. 去伪存真
    • 用户
    • 场景
    • 数据
    • 成本
  3. 优先级判断
    • 用户判断
    • 场景判断
    • 生命周期
    • 需求马斯洛
    • 紧急重要四象限
  4. 产品迭代
    • 版本规划

3.1 需求池

什么事需求池？

需求池就是各式各样需求的集合地，也是灵感和思绪的归宿，更是版本需求的出口

需求池需要明确四个特性

  1. 明确接口人
  2. 明确需求收集时间
  3. 保持更新
  4. 保持反馈

3.3 去伪存真

说白了就是真需求，假需求。这个其实是仁者见仁智者见智的一个东西，不过我们可以根据以下几个标准来衡量

  1. 以用户为中心
    • 以用户为中心，模拟/研究用户使用习惯和思考方式，深挖用户真正的底层需求
  2. 以场景为中心
    • 看看这个需求是不是符合我们的产品，总不能让我的QQ产品来实现点餐
  3. 以数据为均衡标准
    • 通过上线之后的版本和之前得版本对比，来看看这次修改的需求效果如何，是否反而减少了用户量
  4. 以成本为考量
    我们可以如下图评估一下
    • 我们需要找到最第二象限的需求，优先级高

3.3 优先级判断

需求优策略

  • 线上问题优先
  • 核心业务优先
  • 战略需求优先

我们可以把需求分成四个象限，如图所示

  • 我们看一下各个象限的问题

    1. 紧急又重要的事情太多，代表需求管理有问题
    2. 重点放在重要不紧急的事情；
    3. 对于紧急不重要的事情可以做，但不要投入过多的人力和经理
    4. 尽量不要做不重要也不紧急的事

对此我们可以从以下四个部分吧需求分成P0-P3（其实最好是P0-Pn）

  1. 生命周期判断
    根据目前我们产品正处于的生命周期来排序我们需求的优先级
    生命周期如图所示：

  2. 马斯洛需求分析法
    根据马斯洛需求分析法，判断我们的需求属于哪个层次

  3. 用户分析法
    看我们的需求是否接近核心用户，重叠度越高则需求越优先

  4. 场景分析法
    和我们产品契合度高的业务优先级高
    根据使用人数，产生问题频次，还有满意度
